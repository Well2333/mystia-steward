# 东方夜雀食堂料理机制知识库（评分与标签）

更新日期：2026-04-29
适用范围：本仓库后续算法设计、解释文档、规则校验脚本

## 1. 目标与边界

本知识库仅保留可直接用于实现的机制内容，不包含来源说明、证据分级和交叉验证流程。

覆盖内容：

- 稀客评分机制
- 标签生成、相消与禁忌机制
- 特殊顾客的改判与特殊规则（重点含古明地恋）
- 本仓库内部命名与网页展示色彩规范

## 2. 本仓库内部命名与展示规范

### 2.1 评分命名替换

为避免“粉-橙-绿-紫-黑”颜色词在文案与实现中产生歧义，本仓库统一使用：

- 极佳（ExGood）
- 佳（Good）
- 普通（Normal）
- 差（Bad）
- 极差（ExBad）

旧称呼与新命名对应关系：

- 粉 -> ExGood
- 橙 -> Good
- 绿 -> Normal
- 紫 -> Bad
- 黑 -> ExBad

### 2.2 网页展示色彩规范

网页标签底色保持颜色语义，但内部逻辑名一律使用上面的五级命名。

建议标准色：

- ExGood：粉色，`#EC4899`
- Good：橙色，`#F59E0B`
- Normal：绿色，`#22C55E`
- Bad：紫色，`#A855F7`
- ExBad：黑色，`#111827`

### 2.3 相消标签颜色规范

所有“被相消/被覆盖”的标签，统一使用棕色作为标准色。

建议标准色：

- Cancelled Tag（相消标签）：棕色，`#8B5E3C`

建议展示方式：

- 棕色底或棕色文字
- 同时加删除线，表示该标签不参与最终有效标签判定

## 3. 稀客评分机制（实现口径）

### 3.1 点单结构

每次稀客点单包含两项要求：

- 一项料理标签要求
- 一项酒水标签要求

### 3.2 基础计分

- 满足一个顾客喜好标签：`+1`
- 命中一个顾客厌恶标签：`-1`
- 料理与酒水都参与总分计算

### 3.3 五级评价判定（仓库标准）

按以下顺序判定（从上到下命中即停止）：

1. ExGood：总分 `>= 4` 且同时满足料理要求与酒水要求
2. Good：总分 `>= 3` 且至少满足一项点单要求
3. Normal：总分 `>= 2` 且至少满足一项点单要求
4. Bad：总分 `>= 1` 但未达到上述条件
5. ExBad：总分 `<= 0`

说明：

- 该命名用于本仓库逻辑与文档。
- 网页层仅做颜色映射，不使用颜色词作为逻辑字段名。

### 3.4 默认酒水假设与加料上限（节省食材）

由于无法预先确定玩家最终采用的“菜品 + 酒水”组合，本仓库在稀客料理推荐中采用如下默认假设：

- 默认酒水已满足点单酒水要求
- 默认酒水贡献固定 `1` 分

据此，料理侧仅追求 `3` 分即可达到 ExGood 所需总分，不再为料理分数超过 `3` 继续加料。

实现约束：

- 料理搜索优先目标：在当前“最多加料”限制内尽量达到 `foodScore >= 3`；达到后停止继续加料，但最终 `foodScore` 与评级仍按实际有效标签分计算（不做封顶）
- 在达到目标分前提下，优先更少加料数量
- 当多个食材可提供同一目标标签时，优先选择持有数更高者；持有数相同则优先价格更低者
- 当以上条件都相同时，再比较额外成本
- 加料展示标注用途标签：`+食材(标签1,标签2)`

该策略的目的：

- 降低“为无效超分追加食材”带来的浪费
- 提供更稳健、可复现的推荐结果

## 4. 标签机制

### 4.1 标签来源

最终料理标签由三部分合并：

- 料理自带标签
- 加料带入标签
- 动态标签（价格、总食材数、流行）

### 4.2 动态标签

- 菜价 `< 20`：追加“实惠”
- 菜价 `> 60`：追加“昂贵”
- 总食材数 `>= 5`：追加“大份”
- 命中当期流行喜爱：追加“流行喜爱”
- 命中当期流行厌恶：追加“流行厌恶”

### 4.3 标签相消（覆盖）

当前按有向覆盖处理，强标签覆盖弱标签，弱标签被标记为相消（棕色 + 删除线）：

- 肉 覆盖 素
- 大份 覆盖 小巧
- 饱腹 覆盖 下酒
- 重油 覆盖 清淡
- 灼热 覆盖 凉爽

### 4.4 禁忌与黑暗物质

存在禁忌标签组合。命中禁忌时应视为无效推荐组合并在推荐阶段剔除，避免进入可选结果。

## 5. 特殊顾客机制（重点补充）

对古明地恋上 `无意识妖怪慕斯` 会触发最低评价（应作为硬性规避项）。

实现建议：

- 在推荐层加入“恋恋禁用菜”硬过滤，优先级高于评分计算。

## 6. 存档导入映射约束

### 6.1 料理流行标签

- 存档导入优先读取 `playerPartial.popLikeFoodTags` 与 `playerPartial.popHateFoodTags`；若旧存档仍放在 `storagePartial`，则按兼容路径回退读取。
- 当前仓库仍采用“单个料理流行标签 + 正/负方向”的模型，不做多标签叠加。
- 实际存档中的料理流行数组保存为数字 tag id；仓库按现有料理标签顺序把数字 id 映射回标签字符串。
- 若存档数组里存在多个值，只取第一个能映射到当前料理标签集合的有效值；没有有效值则按未设置处理。
- 存档导入后会强制保证“流行喜好”和“流行厌恶”互斥；若检测到喜好标签，则厌恶标签会被清空。
- 导入成功后，结果直接写入全局状态 `popularFoodTag` 与 `popularHateFoodTag`，由普客与稀客推荐共同消费。

### 6.2 暂不接入的存档字段

- `popLikeBevTags`、`popHateBevTags` 当前只做解析保留，不驱动 UI、推荐或配置导出。
- `rewindMode` 当前只做解析保留，不影响现有推荐与筛选逻辑。
- 本轮没有新增配置字段，因此配置导出版本保持不变。

### 6.3 联动稀客显隐

- 存档导入优先读取 `playerPartial.collabStatus`；若旧存档仍放在 `storagePartial`，则按兼容路径回退读取。
- `rewindMode` 同样优先读取 `playerPartial.rewindMode`，兼容回退到 `storagePartial.rewindMode`。
- 全部联动角色与 DLC9（MetaMystia 模组）角色默认加入 `rareHiddenCustomerIds`。
- 存档导入只会根据已开启的联动 key 与 DLC 激活状态，解除这些角色的隐藏状态；不会再通过 `rareExtraCustomerIds` 让它们跨地区强行显示。
- 已实现映射：`MC_Gensokyo -> [30]`、`3FARIES_Collab -> [31]`、`TBS_Kokoro -> [41]`、`TBC2_Collab -> [36, 37, 38]`。
- 语义对应：`30=萌澄果`、`31=蹦蹦跳跳的三妖精`、`41=秦心`、`36/37/38=冴月麟/立空汐/时焉侑`。
- `THYG` 视为“已知空映射”的合法 key：导入时允许出现，但不会增删任何角色。
- DLC9（MetaMystia 模组）角色在未检测到对应 DLC 激活时保持默认隐藏。
- 未知 `collabStatus` key 一律忽略，避免把未核证 key 错误映射到现有角色。

## 7. 本仓库落地约束（供后续开发直接使用）

- 逻辑层只使用 ExGood/Good/Normal/Bad/ExBad，不使用颜色词做枚举值。
- 展示层通过颜色映射渲染评级标签底色。
- 相消标签颜色固定为棕色，并必须可视化删除线。
- 推荐输出应同时保留：原始得分、改判后等级、触发的特殊规则名称，便于排错。
